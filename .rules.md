# 項目規則 (Rules)\n\n## 一般規則\n\n### 語言\n✅ 所有與此項目相關的 Copilot 互動均使用**繁體中文**\n- Chat 消息使用繁體中文\n- 代碼註解使用繁體中文\n- 文檔和說明使用繁體中文\n- 代碼本身（變數名、函數名等）保持英文\n\n### 代碼質量\n✅ 代碼必須符合以下標準：\n- 變數名明確有意義\n- 函數長度不超過50行\n- 複雜邏輯需要添加註解\n- 使用 const 而非 var\n- 避免深度嵌套 (最多3層)\n\n### 文件組織\n✅ 遵循以下結構：\n```\n/server          - 後端代碼\n  /routes        - API 路由\n  /db            - 數據庫相關\n/dist            - 前端編譯產物\n/scripts         - 工具腳本\n/.claude         - 文檔和配置\n```\n\n### 提交規範\n✅ Git 提交遵循：\n- 使用 `ck \"description\"` 提交\n- 一個提交一個功能\n- 使用英文描述但可接受中文\n\n## 技術規則\n\n### 前端開發\n✅ 必須遵守：\n- React 組件使用 Hooks\n- TypeScript 類型標註\n- CSS 使用 Tailwind 或組件內樣式\n- Props 驗證使用 TypeScript\n- 異步操作使用 async/await\n\n### 後端開發\n✅ 必須遵守：\n- 使用 Express 路由\n- 參數驗證在路由處理器開始\n- 錯誤返回統一格式: `{ error: \"message\" }`\n- 成功返回統一格式: `{ data: {...} }` 或直接返回數據\n- SQL 使用參數化查詢防止注入\n\n### 數據庫\n✅ 必須遵守：\n- SQLite 表名使用 snake_case\n- 列名使用 snake_case\n- 主鍵命名為 id\n- 時間戳使用 ISO 8601 格式\n\n## 文檔規則\n\n### Markdown 文件\n✅ 所有 Markdown 文件必須：\n- 使用繁體中文\n- 有清晰的標題層級結構\n- 代碼塊標註語言\n- 重要信息使用粗體或列表突出\n- 包含更新日期\n\n### 代碼註解\n✅ 代碼註解規則：\n```javascript\n// 單行註解使用兩個斜線\n/* 多行註解\n   使用斜線星號 */\n\n/**\n * 函數說明註解\n * @param {type} paramName - 參數說明\n * @returns {type} 返回值說明\n */\nfunction myFunc(paramName) {\n  // 複雜邏輯需要說明\n}\n```\n\n## API 設計規則\n\n### 端點命名\n✅ RESTful API 應遵循：\n- GET `/api/resource` - 獲取列表\n- GET `/api/resource/:id` - 獲取單個\n- POST `/api/resource` - 建立\n- PUT/PATCH `/api/resource/:id` - 更新\n- DELETE `/api/resource/:id` - 刪除\n\n### 參數命名\n✅ 查詢參數使用 camelCase：\n- `?year=2024`\n- `?beadName=tCREA`\n- `?sortBy=date&order=desc`\n\n### 響應格式\n✅ 所有響應應包含：\n```javascript\n// 成功\n{ success: true, data: {...} }\n\n// 錯誤\n{ success: false, error: \"錯誤說明\" }\n\n// 或直接返回數據/錯誤\n```\n\n## 審查規則 (Code Review)\n\n### 檢查項目\n✅ 每次提交審查以下：\n1. 代碼風格一致性\n2. 是否有明顯的 bug\n3. 性能是否有問題\n4. 是否遵循命名規範\n5. 異常處理是否完善\n6. 是否有安全隱患\n\n### 評論格式\n✅ 代碼審查評論：\n- 説明問題位置\n- 解釋為什麼是問題\n- 提供改進建議\n- 使用繁體中文\n\n## 性能規則\n\n### 前端性能\n✅ 必須遵守：\n- 使用虛擬列表渲染大型列表\n- 圖片懶加載\n- 代碼分割按需加載\n- 避免不必要的 re-render\n\n### 後端性能\n✅ 必須遵守：\n- 查詢使用索引\n- 避免 N+1 查詢問題\n- 使用分頁處理大量數據\n- 緩存頻繁查詢\n\n## 安全規則\n\n✅ 必須遵守：\n- SQL 使用參數化查詢\n- 輸入驗證在服務器端\n- 敏感信息不要提交到 Git\n- CORS 正確配置\n- 密碼使用 hash 存儲\n\n## 異常規則\n\n### 錯誤處理\n✅ 必須：\n- 使用 try-catch 包裹可能出錯的代碼\n- 提供有意義的錯誤消息\n- 記錄錯誤到日誌\n- 不要吞掉錯誤（至少記錄）\n\n### 日誌規則\n✅ 日誌應包含：\n- 時間戳\n- 日誌級別 (INFO/WARN/ERROR)\n- 清晰的消息\n- 相關的上下文信息\n\n---\n\n**最後更新**: 2026-04-24  \n**語言**: 繁體中文  \n**版本**: 1.0\n
